## The Right Architecture From the Start

Most front-end frameworks were built for a different era and carry the weight of decisions that made sense in 2013. Natura11y was designed after CSS Cascade Layers, CSS Custom Properties, logical properties, `:is()`, and `:has()` were all available in every modern browser. The result is a framework that is smaller, more overridable, and more capable than those built on legacy foundations.

</TextBlock>

</Section>

<Section id="one-stylesheet">

<TextBlock>

## One Stylesheet. Vanilla HTML and React.

Bootstrap React is a community fork that lags behind the official release and requires a different stylesheet. Tailwind has no components at all. Natura11y ships one stylesheet — `natura11y.min.css` — that works identically whether you are writing plain HTML or building React components.

No separate React stylesheet. No `dark:` utility prefixes. No build-time compilation required to see your styles. The same file, the same tokens, the same semantic patterns in every context.

This is not a coincidence of implementation — it is a core architectural decision. The React component library imports the same utilities the CSS framework uses. The design system is the same system at every layer.

</TextBlock>

</Section>

<Section id="cascade-layers">

<TextBlock>

## @layer Means Your Overrides Always Win

Bootstrap, USWDS, and most older frameworks fight specificity constantly. They solve it with `!important`, increasingly specific selectors, or high-specificity utility classes designed to override component styles. The result is a codebase where changing anything requires understanding the entire specificity chain.

Natura11y wraps every component in `@layer component` and every utility in `@layer utility`. Any CSS you write outside of a layer — meaning any custom CSS in your own project — automatically wins over both. No `!important`. No specificity math. The override model is architecturally correct.

Your customizations are never fighting the framework. They are always on top.

</TextBlock>

</Section>

<Section id="theming">

<TextBlock>

## Theming is a Cascade, Not a Config File

Bootstrap's theming is largely compile-time. You override Sass variables, rebuild, and ship. Dark mode required a second compiled stylesheet for years. Tailwind requires `dark:` prefixes on individual elements throughout your HTML.

Natura11y's theming works at runtime through CSS Custom Property inheritance. Add `theme-dark` to a section and every element inside — headings, body text, borders, links, buttons — adapts to that context automatically. Add `theme-primary` to a card and only that card changes. The cascade handles it.

<CodeBlock
  code={`
<!-- The entire section adapts. No per-element classes needed. -->
<section class="theme-dark padding-4">
    <h2>This heading is already the right color.</h2>
    <p>So is this paragraph, and every link and button inside.</p>
</section>
`}
/>

This also means dark mode is a single variable swap — not a second stylesheet, not a utility prefix on every element.

</TextBlock>

</Section>

## The Entire Type Scale Lives in Two Variables

Every font size in Natura11y — body sizes, all six heading levels, banner headline — is derived from two custom properties:

- `--base-font-size` — the smallest size in the scale
- `--type-scale` — the ratio between each step

Change either variable at a breakpoint and the entire typographic hierarchy recalculates in one declaration. Make the scale tighter on mobile. Open it up on desktop. Adjust the base size for a denser reading environment. The whole system responds.

This is fundamentally different from `clamp()`, which controls the range of a single property. Two variables here control the position and ratio of the entire hierarchy simultaneously.

</TextBlock>

<Section id="logical-properties">

<TextBlock>

## Right-to-Left Support is Architectural, Not an Afterthought

Bootstrap added RTL support in v5 by shipping a second compiled stylesheet. Tailwind requires explicit `rtl:` utility prefixes on individual elements.

Natura11y uses CSS Logical Properties throughout — `padding-inline`, `margin-block`, `border-inline-end`, `inset-inline-start`. Every spacing, border, and layout property is direction-aware by default. Right-to-left language support requires no additional configuration, no second stylesheet, no utility prefixes. It works because the foundation is correct.

</TextBlock>

<Section id="accessibility">

<TextBlock>

## Accessible by Design, Not by Checklist

Accessibility in Natura11y is not a layer applied after components are built — it is how components are built.

- Focus traps in modals and flyouts use the `inert` attribute, the modern standard that actually removes elements from the accessibility tree rather than just hiding them visually
- The collapse and accordion patterns manage `aria-expanded`, focus return on close, and <kbd>Escape</kbd> key behavior in a single, consistent implementation
- `[aria-current="page"]` drives pagination active state — the semantic attribute does the work instead of a visual-only class
- Flyout menus use proper `role="dialog"`, `aria-modal`, and distinct `aria-label` values on the dialog and the navigation inside it — a distinction most implementations miss
- Screen reader utilities — `.screen-reader-only`, `.focusable-only`, `.skip-links` — are first-class, not optional add-ons

Every interactive component is keyboard navigable and works with assistive technology by default.

</TextBlock>

<Section id="minimal-javascript">

<TextBlock>

## Minimal JavaScript With No Dependencies

Natura11y's JavaScript is a small set of focused utility files — focus management, overlay behavior, keyboard navigation — with zero external dependencies. No jQuery. No Popper.js. No component registry or framework initialization.

The flyout works because of focused JavaScript and CSS doing the heavy lifting together. `:has()` drives visual state that previously required JavaScript. `inert` handles accessibility concerns that used to require event listener management. Modern CSS and HTML carry more weight so the JavaScript surface area stays small.

Compare this to Bootstrap's JS bundle, which requires Popper.js for positioning and ships a component initialization system on top. Less JavaScript means faster load times, fewer attack surfaces, and easier debugging.

</TextBlock>

<Section id="ships-complete">

<TextBlock>

## Ships What Others Don't

Most frameworks stop at buttons, forms, and modals. Natura11y ships the components developers spend project after project rebuilding from scratch.

**Backdrop** — Full-bleed hero layouts with image and video backgrounds, built-in overlay support, responsive stacking. Almost no framework provides this.

**Lightbox** — A production-ready, accessible lightbox with gallery support, keyboard navigation (<kbd>Arrow</kbd> keys, <kbd>Escape</kbd> to close), and focus management. Essentially no major framework ships this.

**Flyout with drill-down navigation** — A mobile flyout menu that supports nested panel navigation, `inert`-based panel management, and a `ResizeObserver` that automatically closes it when the viewport changes. This is the component developers dread building.

**Track (Carousel)** — With responsive column counts, peeking, and `prefers-reduced-motion` support built in — not bolted on.

These are not edge cases. They are the components every real production application needs.

</TextBlock>

## Truly Independent. No Corporate Agenda.

Bootstrap was created at Twitter. Material Design is Google's design language. USWDS is maintained by the U.S. federal government. Each of those frameworks — however good — makes decisions shaped by the organization behind it. Features get prioritized based on internal roadmaps. Design opinions reflect the company's product aesthetic. The framework serves the sponsor first.

Natura11y has no corporate owner. It is not a side project of a company's design system that was open-sourced as a marketing move. It was built by developers solving real problems on real projects, with no stakeholder pushing it in a direction that serves a business interest.

That independence shows in the decisions. Accessibility is prioritized because it is right, not because a compliance team required it. Components are included because projects actually need them, not because they fill a slide deck. The architecture is opinionated toward correctness, not toward compatibility with a legacy codebase that can't be changed.

Open source at its best is built this way — by people who use what they make.

</TextBlock>

<Section id="small-codebase">

<TextBlock>

## More Capable. Fewer Lines.

Natura11y achieves feature parity with Bootstrap — and surpasses it in several areas — in significantly less CSS. The reasons are architectural:

- `@extend` shares patterns across components without duplicating rules in the compiled output
- `:is()` and `:has()` reduce selectors that previously required separate rule sets
- CSS Custom Properties eliminate Sass variable compilation for anything that needs to be runtime-adjustable
- `@layer` eliminates defensive utility classes written at high specificity just to override component styles

The stylesheet is readable. Every component file has its variables, then its styles, in one place. Nothing is generated from obscure Sass maps. Nothing requires tracing through five abstraction layers to understand why something looks the way it does.

For a developer auditing a framework before adopting it, that matters.

</TextBlock>
